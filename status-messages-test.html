WCAG 4.1.3 Requirements:

  • Status Messages: Must be programmatically determinable through role or properties
  • No Focus Change: Status must be announced WITHOUT moving focus to the message
  • role="alert": For urgent/important messages (aria-live="assertive" implicit)
  • role="status": For non-urgent status updates (aria-live="polite" implicit)
  • aria-live: Creates live region for dynamic content announcements
  • aria-atomic: Announces entire region vs just changed content
  • Level: AA

Key ARIA Attributes:

Attribute/Role Use Case Behavior
role="alert" Error messages, urgent notifications Interrupts immediately (assertive)
role="status" Success messages, counters, search results Waits for pause in speech (polite)
aria-live="polite" Non-urgent updates Announced when convenient
aria-live="assertive" Critical/time-sensitive messages Interrupts immediately
aria-atomic="true" Progress updates, counters Reads entire region, not just changes

Test Structure:

Each test case shows the SAME message type side-by-side:

  • Left (Red/FAIL): Visual-only message without ARIA roles/attributes
  • Right (Green/PASS): Same visual with proper ARIA for screen reader announcement

Common Violations:

  • Form Errors: Validation messages without role="alert"
  • Success Messages: Confirmation text without role="status"
  • Loading States: Spinners/progress without aria-live region
  • Toast Notifications: Popup messages without role="alert"
  • Live Counters: Dynamic counts without aria-live="polite"
  • Progress Bars: Upload/download status without announcements

Screen Reader Impact:

  • Without ARIA: User only sees visual change, never hears update
  • With role="alert": Screen reader immediately interrupts to announce
  • With role="status": Screen reader announces at next pause
  • User Experience: Blind users miss critical feedback (errors, confirmations, progress)

AI Testing Capabilities:

  • Can detect status/error message divs missing role="alert" or role="status"
  • Can identify toast/notification components without live region attributes
  • Can detect progress indicators without aria-live announcements
  • Can identify dynamic content regions that should be live regions
  • Can check for proper aria-atomic usage on counters and status text
