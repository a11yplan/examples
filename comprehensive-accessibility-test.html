2. Color Contrast Issues Violation WCAG 1.4.3 Level AA

WCAG 1.4.3 Contrast (Minimum): Text must have a contrast ratio of at least 4.5:1 (3:1 for large text).

Examples:

This text has insufficient contrast (color: #d3d3d3 on white background). Contrast ratio: ~1.6:1

This text has very low contrast (color: #e8e8e8 on #f0f0f0 background). Contrast ratio: ~1.2:1

Why it matters: Users with low vision or color blindness cannot read text with insufficient contrast.
How to fix: Use colors that meet minimum contrast ratios. Test with tools like WebAIM Contrast Checker.

Examples of violations:

Why it matters: Screen reader users cannot understand what information to enter without proper labels.
How to fix: Use <label for="input-id"> or wrap inputs with <label> tags. Add autocomplete attributes for personal info.

Examples:

5. Duplicate IDs Violation WCAG 4.1.1 Level A

WCAG 4.1.1 Parsing: ID attributes must be unique on a page.

Examples:

First element with duplicate ID
Second element with duplicate ID (violation)
Why it matters: Duplicate IDs break assistive technology navigation and form label associations.
How to fix: Ensure all IDs are unique across the page.

6. Improper Heading Hierarchy Violation WCAG 1.3.1 Level A

WCAG 1.3.1 Info and Relationships: Heading levels should not be skipped to maintain proper document structure.

Example:

This section has an h2, but then skips to h4:

This h4 skips h3 (heading hierarchy violation)

Why it matters: Screen reader users navigate by headings; skipped levels confuse document structure.
How to fix: Use heading levels in order (h1, h2, h3) without skipping.

Correct Example:

Proper heading hierarchy maintains sequential order

7. Table Without Proper Headers Violation WCAG 1.3.1 Level A

WCAG 1.3.1 Info and Relationships: Data tables must use proper header markup (<th>) to define column/row headers.

Example:

Name Age City
John 30 New York
Jane 25 Boston
Why it matters: Screen reader users cannot understand table structure without proper headers.
How to fix: Use <th> for header cells and add scope attribute.

Correct Example:

Name Age City
John 30 New York
Jane 25 Boston

8. ARIA Misuse and Invalid Attributes Violation WCAG 4.1.2 Level A

WCAG 4.1.2 Name, Role, Value: ARIA attributes must be valid and used correctly.

Examples:

Element with invalid ARIA attribute
Element with invalid ARIA role
List item without role="listitem"
Why it matters: Invalid ARIA causes unpredictable assistive technology behavior.
How to fix: Use valid ARIA attributes, roles, and values according to ARIA specification.

9. Keyboard Accessibility Issues Violation WCAG 2.1.1, 2.4.7 Level A/AA

WCAG 2.1.1 Keyboard: All functionality must be operable through keyboard.
WCAG 2.4.7 Focus Visible: Keyboard focus indicator must be visible.

Examples:

Clickable div without keyboard support (try pressing Enter)
Focusable element with no visible focus indicator
Why it matters: Keyboard-only users cannot access functionality without proper keyboard support.
How to fix: Use native interactive elements (button, a, input), ensure visible focus indicators, avoid positive tabindex values.

Correct Example:

10. Missing Landmark Regions Violation WCAG 1.3.1 Level A

WCAG 1.3.1 Info and Relationships: Page regions should be identified with HTML5 landmarks or ARIA roles.

Example:

This main content area should be wrapped in a <main> tag or role="main".

The navigation above should use <nav> or role="navigation".

Why it matters: Screen reader users rely on landmarks to navigate page structure quickly.
How to fix: Use semantic HTML5 elements: <header>, <nav>, <main>, <footer>, <aside>.

11. Non-descriptive Link Text Violation WCAG 2.4.4 Level A

WCAG 2.4.4 Link Purpose (In Context): Link purpose should be clear from link text alone.

Examples:

Click here to learn more about accessibility.
Read more
Here is important information.
Why it matters: Screen reader users often navigate by links list; non-descriptive text provides no context.
How to fix: Use descriptive link text that makes sense out of context.

Correct Example:

Read the complete accessibility guidelines

12. Missing or Incorrect Language Attributes Violation WCAG 3.1.1, 3.1.2 Level A

WCAG 3.1.1 Language of Page: The default language of the page must be programmatically determinable.
WCAG 3.1.2 Language of Parts: When language changes within content, it must be marked.

Examples:

Este contenido está en español pero no tiene atributo lang.

Ce contenu est en français mais n'a pas d'attribut lang.

Why it matters: Screen readers need language info to use correct pronunciation.
How to fix: Add lang attribute: <p lang="es">Spanish text</p>

Correct Example:

Este contenido está correctamente marcado con lang="es".

WCAG 3.3.1 Error Identification: Errors must be identified and described to users.
WCAG 3.3.3 Error Suggestion: Suggestions for fixing errors should be provided.

Example:

Why it matters: Users need clear feedback about errors and how to fix them.
How to fix: Add aria-invalid, aria-describedby, and clear error messages.

15. Media Without Captions or Transcripts Violation WCAG 1.2.2, 1.2.3 Level A

WCAG 1.2.2 Captions (Prerecorded): Captions must be provided for prerecorded audio in video.
WCAG 1.2.3 Audio Description or Media Alternative: Provide alternative or audio description.

Example:

Why it matters: Deaf/hard of hearing users cannot access audio content without captions.
How to fix: Add <track> element with captions and provide transcripts.

16. Modal Dialog Without Proper Focus Management Violation WCAG 2.4.3 Level A

WCAG 2.4.3 Focus Order: Modal dialogs must trap focus and return it on close.

Example:

Modal Without Focus Management

This modal doesn't trap focus or have proper ARIA attributes.

Why it matters: Keyboard users can tab out of modal, losing context and access to controls.
How to fix: Implement focus trap, add role="dialog", aria-modal="true", manage focus on open/close.

20. Auto-updating Content Without Pause Control Violation WCAG 2.2.2 Level A

WCAG 2.2.2 Pause, Stop, Hide: Moving, blinking, scrolling, or auto-updating content must have pause/stop controls.

Example:

This content updates every 5 seconds without user control
Why it matters: Auto-updating content can distract users with cognitive disabilities and interrupt screen readers.
How to fix: Provide pause/stop button or allow user to control update frequency.
