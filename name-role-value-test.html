WCAG 4.1.2 Requirements:

  • Name: Accessible name (aria-label, aria-labelledby, or associated label)
  • Role: Component role (button, checkbox, link, etc.)
  • Value: Current value/state (aria-checked, aria-pressed, value attribute)
  • States/Properties: Must be programmatically settable and announced
  • Use native HTML controls when possible (button, input, etc.)
  • Custom controls need proper ARIA attributes
  • Level: A

Test Structure:

Each test case shows the SAME control side-by-side:

  • Left (Red/FAIL): Missing ARIA attributes (role, label, state)
  • Right (Green/PASS): Proper ARIA implementation

Common Violations:

  • Custom Buttons: Divs/spans styled as buttons without role="button" and tabindex
  • Custom Checkboxes: Divs without role="checkbox" and aria-checked
  • Icon Links: Links with only icons/emojis, no accessible name
  • Unlabeled Inputs: Form fields relying only on placeholder text
  • Toggle Buttons: State changes without aria-pressed, aria-expanded, etc.

Screen Reader Impact:

  • Missing Role: Control not identified correctly (just "group" or "clickable")
  • Missing Name: Purpose unknown (announces only role without label)
  • Missing State: State changes not announced (user doesn't know if toggle is on/off)
  • Unlabeled Inputs: Placeholder disappears on focus, cannot be retrieved

  • Can identify custom controls (non-native) that need ARIA attributes
  • Can detect icon-only links/buttons that need aria-label
  • Can identify form inputs without associated labels
  • Can detect toggle/state controls missing aria-pressed/aria-checked
