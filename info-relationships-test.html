WCAG 1.3.1 Requirements:

  • Information and relationships must be programmatically determinable
  • Use semantic HTML elements (table, h1-h6, ul/ol/li, fieldset/legend)
  • Structure conveyed visually must also be in the code
  • ARIA can supplement but shouldn't replace semantic HTML
  • Level: A

Test Structure:

Each test case shows the SAME content side-by-side:

  • Left (Red/FAIL): Visual styling without semantic markup
  • Right (Green/PASS): Proper semantic HTML elements

Common Violations:

  • Div Tables: Using CSS to create table appearance without <table> elements
  • Styled Divs as Headings: Bold/large text without <h1>-<h6> elements
  • Links in Divs (Footer Pattern): Navigation links without <ul>/<ol>/<li> elements - very common in footers
  • Ungrouped Form Controls: Radio/checkbox groups without <fieldset>/<legend>

Screen Reader Impact:

  • Tables: Cannot navigate by cell, row, or column without semantic markup
  • Heading Hierarchy: Cannot navigate by heading level, cannot understand document structure/outline - critical for long documents
  • Link Lists: Cannot know how many links, cannot skip the group - especially problematic in footers with many links
  • Form Groups: Context (legend) not announced with each control

AI Testing Capabilities:

  • Can recognize visual table patterns (rows, columns, headers) even without semantic markup
  • Can identify heading hierarchies by analyzing font sizes, weights, and spacing patterns
  • Can detect groups of related links (footer navigation) that should be lists
  • Can identify form control groupings that need fieldset/legend

Real-World Examples:

  • Div Tables: Often seen in legacy code or CSS-heavy frameworks
  • Styled Div Headings: Common when designers use exact font sizes instead of semantic levels
  • Footer Links in Divs: Extremely common - developers often forget lists can have list-style:none
  • Ungrouped Radios: Frequent in forms built without accessibility in mind
