WCAG 4.1.3 Status Messages - Test Page

Testing: Status messages must be programmatically determinable through role or properties so that assistive technologies can present them to users without receiving focus.

Test Case 1: Form Validation Error Messages

FAIL: Visual-Only Error Message

⚠ Please enter a valid email address

Error message appears visually but is NOT announced to screen readers - no role="alert" or aria-live region.

<!-- FAIL: No ARIA for status message --> <div class="error-visual"> ⚠ Please enter a valid email address </div> <!-- Screen reader users won't hear this error -->

PASS: Error with role="alert"

⚠ Please enter a valid email address

Error message uses role="alert" (equivalent to aria-live="assertive") - screen reader immediately announces the error.

<!-- PASS: role="alert" announces immediately --> <div id="email-error" role="alert"> ⚠ Please enter a valid email address </div> <!-- Screen reader: "Alert: Please enter a valid email address" -->

WCAG 4.1.3 Status Messages (Level AA):

In content implemented using markup languages, status messages can be programmatically determined through role or properties such that they can be presented to the user by assistive technologies without receiving focus.

Key ARIA Roles and Properties:

Technique Behavior Use Case
role="alert" Immediate announcement (assertive) Error messages, urgent warnings
role="status" Polite announcement (at pause) Success messages, search results, updates
aria-live="assertive" Immediate announcement Same as role="alert"
aria-live="polite" Polite announcement (at pause) Loading states, character counts, non-urgent updates
aria-atomic="true" Announce entire region, not just changes Cart counts, timers where context matters

Common Violations:

  • Visual-only status: Messages appear on screen but have no ARIA roles or live regions
  • Missing role="alert": Error messages not announced immediately
  • Missing role="status": Success/confirmation messages not announced
  • No aria-live: Dynamic content changes (counts, loading states) not communicated
  • Wrong politeness level: Using assertive for non-urgent updates (annoying) or polite for errors (may be missed)

Best Practices:

  • Use role="alert" for errors and urgent messages that need immediate attention
  • Use role="status" for success messages, search results, and general updates
  • Use aria-live="polite" for loading indicators and character counts
  • Add aria-atomic="true" when the full context is needed (e.g., "3 items" not just "3")
  • Don't overuse assertive announcements - they interrupt the user
  • Keep status messages concise and meaningful
